FaRM

发表于 2022-10-20 更新于 2022-10-21 分类于 6.824 阅读次数：

6.824 的第九篇论文是 FaRM——一个通过为分布式事务提供序列化、高性能、持久性和高可用性的主存分布式计算平台。

FaRM 利用以下两种硬件趋势做到这一点：

  • 非易失 DRAM（通过将 DRAM 写入 SSD 实现）
  • 快速 RDMA

这消除了存储和网络瓶颈，却带来了新的 CPU 瓶颈。FaRM 通过遵循以下三个原则来解决这一 bottleneck：

  • 减少消息量

    FaRM 允许事务与存储对象跨机器分布；

    使用具有 replica 的垂直 Paxos 以及直接与 Primary/Backup 通信的非复制协调器；

    使用乐观并发控制和四阶段提交协议(4PC: lock, validation, commit backup, commit primary)。

  • 使用单侧 RDMA 进行读写，而不是传递消息

    不使用远程 CPU，且避免了大部分 local CPU 开销。但这样一来传统的基于 CPU 的 failure-recovery 就失效了，需要考虑新的 failure-recovery protocol：

    1. precise membership：FaRM servers 不根据它们的 lease 到期而拒绝传入的请求，因为请求由不支持 lease 的网卡(NIC)提供服务；使用 precise membership来确保机器同意当前的成员配置，并只向成员机器发送单侧操作；
    2. reservations：因为事务记录被写入参与者日志而不涉及 CPU，FaRM 使用 reservations 来确保在开始 commit 之前，日志中有足够空间用于 commit 和 truncate 事务。

  • 有效利用并行性

    它将每个状态的恢复均匀地分布在集群中，并在每台机器之间并行恢复。它使用两个优化来允许事务执行与恢复并行进行：

    1. 事务在锁定恢复阶段之后开始访问受故障影响的数据；
    2. 不受故障影响的事务继续执行而不会阻塞。

硬件趋势

非易失 DRAM

发生电源故障时，分布式不间断电源(UPS)使用电源中的能量将内存中的数据保存到 SSD，从而提高 DRAM 耐用性。

RDMA 网络

FaRM 尽可能使用单侧 RDMA 操作，从而有效减少了 CPU 开销与需传递的消息数量——RPC 需要的消息量是单侧 RDMA 的两倍。

编程模型和架构

FaRM 为应用程序提供了跨越集群中机器的全局地址空间的抽象。每台机器都运行应用程序线程并将对象存储在地址空间中。FaRM API 提供对事务中本地和远程对象的透明访问。应用程序线程可以随时启动事务，并成为事务的协调者。在事务执行期间，线程可以执行任意逻辑以及读取、写入、分配和释放对象。在执行结束时，线程调用 FaRM 来提交事务。

FaRM 事务使用乐观并发控制。更新在执行期间在本地缓冲，并且只在成功提交后对其他事务可见。由于与并发事务的冲突或失败，提交可能会失败。 FaRM 为所有成功提交的事务提供严格的可串行化[35]。在事务执行期间，FaRM 保证单个对象读取是原子的，它们只读取已提交的数据，同一对象的连续读取返回相同的数据，并且由事务写入的对象的读取返回最新写入的值。它不保证读取不同对象的原子性，但在这种情况下，它保证事务不会提交，确保提交的事务是严格可序列化的。这允许我们将一致性检查推迟到提交时间，而不是重新检查每个对象读取的一致性。然而，它增加了一些编程复杂性：FaRM 应用程序必须在执行期间处理这些临时的不一致 [20]。可以自动处理这些不一致[12]。

FaRM API 还提供无锁读取（优化的单对象只读事务）和位置提示，使程序员能够在同一组机器上共同定位相关对象。这些可以被应用程序用来提高性能，如 [16] 中所述。

图 3 显示了一个带有四台机器的 FaRM 实例。该图还显示了机器 A 的内部组件。每台机器在用户进程中运行 FaRM，每个硬件线程都有一个内核线程。每个内核线程都运行一个事件循环，该循环执行应用程序代码并轮询 RDMA 完成队列。

随着机器出现故障或添加新机器，FaRM 实例会随着时间的推移在一系列配置中移动。配置是一个元组，其中i是唯一的、单调递增的64位配置标识符，S是配置中的机器集，F是从机器到故障域的映射预计会独立失败（例如，不同的机架），并且 CM ∈ S 是配置管理器。 FaRM 使用 Zookeeper [21] 协调服务来确保机器就当前配置达成一致并存储它，就像在 Vertical Paxos [25] 中一样。但它不像通常那样依赖 Zookeeper 来管理租约、检测故障或协调恢复。 CM 使用利用 RDMA 快速恢复的高效实现来执行这些操作。每次配置更改时，CM 都会调用一次 Zookeeper 来更新配置。

FaRM 中的全局地址空间由 2 GB 区域组成，每个区域在一个主节点和 f 个备份上复制，其中 f 是所需的容错能力。每台机器在非易失性 DRAM 中存储多个区域，其他机器可以使用 RDMA 读取这些区域。对象总是从包含区域的主副本中读取，如果区域在本地机器上，则使用本地内存访问，如果是远程，则使用单面 RDMA 读取。每个对象都有一个用于并发控制和复制的 64 位版本。区域标识符到其主要和备份的映射由 CM 维护并与区域一起复制。这些映射由其他机器按需获取，并由线程与向主节点发出单侧 RDMA 读取所需的 RDMA 引用一起缓存。

机器联系 CM 以分配新区域。 CM 从一个单调递增的计数器中分配一个区域标识符，并为该区域选择副本。副本选择平衡每台机器上存储的区域数量，受以下约束：有足够的容量，每个副本位于不同的故障域中，当应用程序指定局部性约束时，该区域与目标区域位于同一位置.然后它将准备消息发送到带有区域标识符的选定副本。如果所有副本都报告分配区域成功，则 CM 向所有副本发送提交消息。这个两阶段协议确保映射在使用之前有效并在所有区域副本中复制。

与我们之前基于一致散列的方法相比，这种集中式方法提供了更大的灵活性来满足故障独立性和局部性约束[16]。它还可以更轻松地平衡机器之间的负载并在接近容量的情况下运行。使用 2 GB 区域，我们预计一台典型机器上最多有 250 个区域，因此单个 CM 可以处理数千台机器的区域分配。

每台机器还存储实现 FIFO 队列的环形缓冲区 [16]。它们用作事务日志或消息队列。每个发送者-接收者对都有自己的日志和消息队列，它们物理上位于接收者上。发送者使用单侧 RDMA 写入其尾部将记录附加到日志中。这些写入由 NIC 确认，而不涉及接收器的 CPU。接收器定期轮询日志的头部以处理记录。它在截断日志时延迟更新发送方，允许发送方重用环形缓冲区中的空间。
